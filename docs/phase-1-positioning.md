# Phase 1: 网站定位与策略

## 一、网站定位

**研究笔记型 + 成长档案，带知识花园气质。**

以深度技术长文为主体内容，允许半成品笔记公开迭代，辅以项目展示区支撑求职场景。
网站气质定位为"安静的知识仓库"——不追求流量、不设评论区、不搞社交互动，
专注于内容深度和个人知识体系的积累。

## 二、目标用户

| 优先级 | 用户角色 | 访问目的 | 期望传递的信号 |
|---|---|---|---|
| P0 | 未来的自己 | 检索已沉淀的知识 | 快速回忆技术细节与当时思考 |
| P1 | LLM 方向面试官/团队 | 查看项目 + 代表性文章 | 技术功底扎实，有独立研究品味 |
| P2 | 同领域研究者/同学 | 论文笔记、实验记录 | 技术讨论起点，潜在合作可能 |
| P3 | 搜索引擎带来的路人 | 某篇特定技术文章 | 内容有质量，值得收藏 |

## 三、技术栈

| 层面 | 选择 | 理由 |
|---|---|---|
| 静态站点生成器 | Astro | Content Collections 天然支持多内容类型，Islands Architecture，默认零 JS |
| 样式方案 | Tailwind CSS + 设计令牌 | 通过约束集防止样式熵增，长期维护一致性好 |
| 内容格式 | MDX (Markdown + 组件) | 技术文章需要公式、代码块、自定义组件嵌入 |
| 部署 | GitHub Actions → GitHub Pages | 源码在 main 分支，构建产物推送到 gh-pages 分支 |
| 域名 | dokivot.github.io | 用户/组织站点，无需额外配置自定义域名 |

## 四、内容成熟度模型

允许半成品内容公开，通过成熟度标记区分内容可靠程度：

| 标记 | 级别 | 含义 |
|---|---|---|
| 🌱 | 笔记 (seedling) | 碎片化记录，可能不完整，但信息有用 |
| 🌿 | 草稿 (budding) | 结构完整，尚未精修，可能有疏漏 |
| 🌳 | 成文 (evergreen) | 经过打磨，可对外引用的完整文章 |

此标记出现在文章列表和文章详情页，降低发布心理门槛。

## 五、内容类型定义

| 类型 | 内容定位 | 允许的成熟度 | 路由前缀 |
|---|---|---|---|
| 技术博客 | 深度长文，核心内容支柱 | 仅 🌳 | `/blog/` |
| 学习笔记 | 半成品知识卡片，持续迭代 | 🌱🌿🌳 全部 | `/notes/` |
| 项目展示 | 求职入口页，结构化展示 | 仅 🌳 | `/projects/` |
| 技术文档 | 教程/指南类系统性文档 | 仅 🌳 | `/docs/` |
| 个人介绍 | 静态个人页 | 不适用 | `/about/` |

## 六、发展规划

### 第一阶段：扎根（当前 — 6 个月）

- 建立站点基础框架
- 完成个人介绍页和 3+ 项目展示
- 产出 5-10 篇核心领域深度文章（Transformer、注意力机制、训练技巧等）
- 目标：站点可用，内容骨架成型

### 第二阶段：生长（6 — 18 个月）

- 形成稳定的写作节奏
- 文章覆盖更广主题（RLHF、量化、推理优化、多模态等）
- 笔记区开始积累，形成知识网络
- 目标：站点成为个人知识体系的外化载体

### 第三阶段：沉淀（18 个月以后）

- 内容规模达到一定程度后，增加搜索、相关推荐等功能
- 项目展示区随职业发展更新
- 目标：站点成为领域内一张安静但有力度的名片

## 七、设计原则

1. **内容优先**：排版、公式、代码块的阅读体验是最高优先级
2. **安静克制**：无社交元素（评论区、点赞、阅读量），专注于内容本身
3. **长期可维护**：约束式设计系统，避免样式熵增
4. **低发布门槛**：成熟度模型允许半成品存在，降低心理负担
